Query Profile Operations Deep Dive

Core Operation Types (Memorize for Exam)

  • Scan: Data read from datasource, check for file pruning issues
  • Join: Rows combined from multiple relations, optimize join order
  • Union: Rows concatenated from relations with same schema
  • Shuffle: Data redistributed between executors (most expensive)
  • Hash/Sort: Grouping and aggregation operations
-- Query designed to show interesting profile patterns
-- This will demonstrate all major operation types
WITH regional_sales AS (
  -- SCAN operations: 3 table scans
  SELECT 
    st.region,
    p.category,
    DATE_TRUNC('month', s.date) as month,
    SUM(s.total_amount) as monthly_revenue,
    COUNT(*) as transaction_count
  FROM databricks_clar.silver.sales_enriched s
  -- JOIN operations: 2 INNER JOINs
  INNER JOIN databricks_clar.silver.products_enriched p 
    ON s.product_id = p.product_id
  INNER JOIN databricks_clar.silver.stores_enriched st
    ON s.store_id = st.store_id
  WHERE s.date >= '2025-01-01'
  -- SHUFFLE operation: GROUP BY aggregation
  GROUP BY st.region, p.category, DATE_TRUNC('month', s.date)
),
ranked_performance AS (
  SELECT 
    region,
    category,
    month,
    monthly_revenue,
    -- SHUFFLE operation: Window functions
    LAG(monthly_revenue) OVER (
      PARTITION BY region, category 
      ORDER BY month
    ) as prev_month_revenue,
    RANK() OVER (
      PARTITION BY month 
      ORDER BY monthly_revenue DESC
    ) as revenue_rank
  FROM regional_sales
)
-- SORT operation: Final ORDER BY
SELECT * FROM ranked_performance
ORDER BY month DESC, monthly_revenue DESC;
Bottleneck Analysis:
• File pruning issues → Scan nodes with 0 files pruned
• Expensive joins → Large row counts in join operations
• Shuffle operations → Data movement between nodes
• Enable verbose mode for complete operation details

WITH slow_queries AS (

    query_hash,
    LEFT(query_text, 200) as query_preview,
    user_name,
    compute_name,
    query_start_time,

    AND query_text NOT LIKE 'SHOW%' -- Exclude metadata queries

  query_preview,
  user_name,
  compute_name,
  duration_seconds,
  mb_read,
  estimated_cost_usd,
  CASE 
    WHEN duration_seconds > 300 THEN 'Critical'
    WHEN duration_seconds > 60 THEN 'Warning'
    ELSE 'Normal'
  END as performance_category
FROM slow_queries
ORDER BY duration_seconds DESC
LIMIT 10;
-- Warehouse utilization pattern analysis
WITH hourly_usage AS (
  SELECT 
    compute_name,
    DATE_TRUNC('hour', query_start_time) as hour,
    COUNT(*) as query_count,
    AVG(total_duration_ms) / 1000.0 as avg_duration_seconds,
    SUM(read_bytes) / (1024 * 1024 * 1024) as total_gb_read,
    COUNT(DISTINCT user_name) as unique_users
  FROM system.query.history
  WHERE query_start_time >= CURRENT_DATE - INTERVAL 7 DAYS
    AND total_duration_ms IS NOT NULL
  GROUP BY compute_name, DATE_TRUNC('hour', query_start_time)
),
usage_summary AS (
  SELECT 
    compute_name,
    hour,
    query_count,
    avg_duration_seconds,
    total_gb_read,
    unique_users,
    -- Peak detection
    CASE 
      WHEN query_count > AVG(query_count) OVER (PARTITION BY compute_name) * 1.5 
      THEN 'Peak Hours'
      WHEN query_count < AVG(query_count) OVER (PARTITION BY compute_name) * 0.5 
      THEN 'Low Usage'
      ELSE 'Normal'
    END as usage_category
  FROM hourly_usage
)
SELECT 
  compute_name,
  usage_category,
  COUNT(*) as hours_in_category,
  AVG(query_count) as avg_queries_per_hour,
  AVG(total_gb_read) as avg_gb_per_hour
FROM usage_summary
GROUP BY compute_name, usage_category
ORDER BY compute_name, usage_category;

Query Caching Architecture

Multi-Layer Caching System

Cache Type Scope Lifetime Storage Warehouse Support Key Benefits
SQL UI Cache Per-user 7 days maximum Databricks filesystem All types Dashboard refresh speed
Local Result Cache Per-cluster 24 hours from entry Cluster memory/local All types Identical query acceleration
Remote Result Cache Workspace-wide 24 hours, survives restart Workspace system data Serverless ONLY Cross-warehouse sharing
Disk Cache Local cluster Until cluster restart Local SSD storage All types Data file acceleration
Remote Result Cache is the PRIMARY differentiator for Serverless warehouses. It persists across restarts and is shared across all warehouses in a workspace.

Cache Lookup Hierarchy & Processing Flow

Query Processing Steps

  1. Local Cache Check: Search cluster memory for identical query
  2. Remote Cache Check: Search workspace cache (Serverless only)
  3. Execute Query: Full processing if no cache hit
  4. Store Results: Cache results for future use
-- Cache demonstration with Serverless warehouse

-- Third execution: Remote cache hit (Serverless only!)

Cache States and Performance

  • Cold State: No cached data, full execution required (slowest)
  • Warm State: Disk cache available, faster data reads
  • Hot State: Result cache hit, instant return (fastest)

Cache Invalidation & Control

Automatic Invalidation Triggers

  • Table Updates: Any DML operation (INSERT, UPDATE, DELETE, MERGE)
  • Schema Changes: ALTER TABLE operations
  • Time Expiration: 24-hour lifecycle for result caches, 7 days for UI cache
  • Cluster Restart: Disk cache cleared, remote cache survives (Serverless)
  • Dynamic Functions: CURRENT_TIMESTAMP(), CURRENT_DATE(), etc.

Monitoring Cache Effectiveness

-- Query to monitor cache hit rates and performance
WITH cache_analysis AS (
  SELECT 
    DATE_TRUNC('hour', query_start_time) as hour,
    compute_name,
    COUNT(*) as total_queries,
    -- Identify potential cache hits (very fast queries)
    COUNT(CASE WHEN total_duration_ms < 1000 THEN 1 END) as fast_queries,
    -- Average execution time
    AVG(total_duration_ms) / 1000.0 as avg_duration_seconds,
    -- Data processed
    AVG(read_bytes) / (1024 * 1024) as avg_mb_read
  FROM system.query.history
  WHERE query_start_time >= CURRENT_DATE - INTERVAL 7 DAYS
    AND total_duration_ms IS NOT NULL
    AND compute_name IS NOT NULL
  GROUP BY DATE_TRUNC('hour', query_start_time), compute_name
)
SELECT 
  hour,
  compute_name,
  total_queries,
  fast_queries,
  ROUND((fast_queries * 100.0) / total_queries, 2) as cache_hit_rate_pct,
  avg_duration_seconds,
  avg_mb_read,
  CASE 
    WHEN (fast_queries * 100.0) / total_queries > 50 THEN 'High Cache Utilization'
    WHEN (fast_queries * 100.0) / total_queries > 20 THEN 'Moderate Cache Utilization'
    ELSE 'Low Cache Utilization'
  END as cache_effectiveness
FROM cache_analysis
WHERE total_queries >= 5 -- Focus on meaningful sample sizes
ORDER BY hour DESC, cache_hit_rate_pct DESC;

📈 Monitoring & Profiling (25% weight)
  • Three Layers: Query Profile, Warehouse Monitoring, System Tables
  • Operations: Scan, Join, Shuffle (expensive), Hash/Sort
  • Permissions: Query owner OR CAN MONITOR warehouse
  • Bottlenecks: File pruning, join optimization, shuffle reduction
  • System Tables: query.history, billing.usage for governance
⚡ Query Caching (30% weight)
  • Four Types: UI (7d), Local (24h), Remote (24h, Serverless), Disk (restart)
  • Hierarchy: Local → Remote → Execute (Serverless path)
  • Invalidation: Table updates, schema changes, dynamic functions
  • Control: SET use_cached_result = false (testing only)
  • Optimization: Cache-friendly query design, warming strategies
