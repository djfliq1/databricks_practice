BRONZE LAYER ↓
01 · Bronze
Raw Ingestion — Auto Loader

Goal

Land raw CSV files from the Volume into a Delta table with zero transformation. Bronze is a faithful copy of the source — we don't clean here, we just capture. If something goes wrong upstream, we can always replay from Bronze.

Why Auto Loader Over COPY INTO

  • Auto Loader — Structured Streaming, incremental, checkpoint-backed, scales to millions of files
  • COPY INTO — SQL batch command, manual trigger, no streaming state. Better for one-time migration, not ongoing ingestion
  • Auto Loader gives exactly-once delivery automatically via the checkpoint
  • availableNow trigger used instead of continuous — processes queued files then stops, saving compute cost vs a 24/7 running stream

Resource Decision

  • No schema enforcement at Bronze — inferSchema=true trades a small overhead for flexibility when source evolves
  • @dp.expect (WARN mode) logs quality violations without dropping rows or killing the pipeline — appropriate for raw landing where we want everything in, violations visible
  • Checkpoint stored in the same catalog Volume — keeps state co-located with data
pipeline_products.py · Bronze Layer
# ── BRONZE LAYER ─────────────────────────────────────────────────────
# Raw landing zone. No transforms — faithful copy of source.
# Purpose: replayable source of truth if Silver/Gold need to be rebuilt.

@dp.table(
    name="bronze_products_raw",
    # comment = visible in Unity Catalog lineage UI
    comment="Raw product CSV ingested from bronze volume via Auto Loader",
    table_properties={
        # Deletion vectors = soft deletes (mark rows deleted, defer file rewrite)
        # Reduces write amplification vs immediate file compaction
        "delta.enableDeletionVectors": "true"
    }
)
# WARN mode: log the violation, keep the row.
# At Bronze we want EVERYTHING in — we'll enforce quality at Silver.
# Using expect_or_fail here would kill the pipeline on a single bad row.
@dp.expect("price_not_null",    "price IS NOT NULL")
@dp.expect("product_id_valid", "product_id IS NOT NULL")
def bronze_products_raw():
    return (
        spark.readStream

        # cloudFiles = Auto Loader source format (not the file format)
        # Think of it as: "use the Auto Loader engine"
        .format("cloudFiles")

        # cloudFiles.format tells Auto Loader what the actual files ARE
        # This is separate from .format("cloudFiles") above — common confusion point
        .option("cloudFiles.format",        "csv")
        .option("header",                    "true")

        # inferSchema adds one pass to read the schema from the files.
        # Trade-off: small overhead vs manually maintaining a schema definition.
        # For a 100-row CSV with stable structure, inference is fine.
        .option("inferSchema",               "true")

        # schemaLocation persists the inferred schema between runs.
        # Without this, Auto Loader re-infers on every restart — wasteful.
        .option("cloudFiles.schemaLocation",  "/Volumes/databricks_clar/bronze/schema/products")

        # addNewColumns = if source adds a column, Auto Loader adds it to the table.
        # Alternative: failOnNewColumns (strict) or rescue (quarantine new cols).
        # For a dev/learning pipeline, addNewColumns is the safe default.
        .option("cloudFiles.schemaEvolutionMode", "addNewColumns")

        # FOLDER path — not a file path. Auto Loader watches the directory.
        # Pointing at a specific file breaks incremental tracking.
        .load("/Volumes/databricks_clar/bronze/bronze_volume/products/")

        # Stamp ingestion time on every row at Bronze.
        # Useful for debugging: when exactly did this row land?
        .withColumn("_ingested_at", current_timestamp())
    )
Checkpoint note: In a DAB-deployed pipeline, the checkpoint location is managed automatically by the pipeline — you don't set it manually in the @dp.table decorator. The framework handles exactly-once delivery. In a standalone streaming job outside DLT/SDP, you'd set .option("checkpointLocation", "...") on the writeStream.
STAGING VIEW (intermediate, not published) ↓
02 · Staging
Staging View — Intermediate Transform

Goal

Apply lightweight transforms (uppercase, type cast) between Bronze and Silver without publishing an intermediate table to Unity Catalog. The staging view is an in-pipeline temp step — it exists only during pipeline execution and consumes no storage.

Why a View Over Another Table

  • Zero storage cost — staging views are never materialized to disk. A staging table would write another full copy of the data
  • Keeps the UC namespace clean — analysts won't accidentally query a half-transformed intermediate table
  • Transforms here are simple column operations — not worth the storage and compute overhead of a full materialized table
  • If transforms were expensive (ML scoring, window functions over large windows), a table would be justified to avoid recomputation
pipeline_products.py · Staging View
# ── STAGING VIEW ──────────────────────────────────────────────────────
# In-pipeline only — NOT published to Unity Catalog.
# Use for: lightweight transforms between Bronze and Silver.
# Cost: zero additional storage. Compute: runs inline during pipeline.

@dp.view(name="products_staging_vw")
def products_staging_vw():
    return (
        spark.readStream

        # skipChangeCommits=True is REQUIRED when reading a Delta table
        # that has CDC (update/delete operations) as a streaming source.
        # Without it, update/delete commit markers cause a streaming error.
        # Our Bronze table may get OPTIMIZE or VACUUM operations — this handles that.
        .option("skipChangeCommits", True)

        # Reference the Bronze table defined above by name.
        # Lakeflow resolves this as a pipeline dependency automatically.
        .table("bronze_products_raw")

        # Normalize category to uppercase for consistent downstream joins.
        # "Books" vs "BOOKS" would be two different groups in GROUP BY — bad.
        .withColumn("category", upper(col("category")))

        # Price as double — inferSchema may have read it as string from CSV.
        # Explicit cast ensures downstream AVG/SUM operations work correctly.
        .withColumn("price", col("price").cast(DoubleType()))

        # Explicit processing flag for lineage visibility.
        # When querying Silver, this tells you the row came through the pipeline.
        .withColumn("flag", lit("pipeline_processed"))
    )
SILVER LAYER ↓
03 · Silver
Clean & Enriched — Streaming Table

Goal

Produce a clean, schema-enforced, de-duplicated Silver table that's safe for downstream analytics and ML. Silver is where data quality becomes a hard contract — not just a warning.

Expectation Mode Decision

  • Bronze used @dp.expect (WARN) — capture everything, flag violations
  • Silver uses @dp.expect_or_drop — rows that violate quality rules are silently dropped before they pollute downstream aggregations
  • @dp.expect_or_fail (stop pipeline) is reserved for truly unrecoverable violations — not used here because a few bad rows shouldn't halt the entire pipeline

Resource Decision

  • Streaming table (@dp.table) — not a materialized view — because we're reading from a streaming Bronze source and want incremental processing
  • Reading from the staging view (not Bronze directly) reuses the transform logic already computed — avoids re-reading Bronze twice
  • No repartition — for 100 rows this is unnecessary overhead. At scale (100M+ rows), you'd repartition to match the number of output files you want
pipeline_products.py · Silver Layer
# ── SILVER LAYER ──────────────────────────────────────────────────────
# Clean, enriched, quality-gated. This is the table analysts query.
# Hard quality contract: bad rows are DROPPED, not just flagged.
# Schema is enforced — no inferred types here.

@dp.table(
    name="silver_products_enriched",
    comment="Cleaned, enriched products — primary Silver table for analytics",
    table_properties={
        # Liquid Clustering on category + price — the two columns most
        # commonly used in WHERE clauses and GROUP BY downstream.
        # More flexible than Z-ORDER: idempotent, key changes don't require
        # a full table rewrite. Replaces static partitioning.
        "delta.feature.liquidClustering": "enabled"
    }
)

# expect_or_drop: row fails → row removed → pipeline continues.
# We don't want nulls in product_id — they'd break any JOIN downstream.
# We don't want negative prices — they'd corrupt AVG calculations.
@dp.expect_or_drop("product_id_not_null", "product_id IS NOT NULL")
@dp.expect_or_drop("price_positive",      "price > 0")
@dp.expect_or_drop("category_not_null",   "category IS NOT NULL")

def silver_products_enriched():
    return (
        spark.readStream

        # Required when source is a Delta table with CDC operations.
        # The staging view feeds from Bronze which can receive OPTIMIZE/VACUUM.
        # Without this flag, those commit markers cause a streaming read error.
        .option("skipChangeCommits", True)

        # Read from staging view — not Bronze directly.
        # Staging already applied: uppercase, cast, flag column.
        # Reading from staging avoids duplicating that transform logic here.
        .table("products_staging_vw")

        # Add a price tier column — drives downstream Gold categorization.
        # Using when/otherwise instead of a UDF: SQL-native, Catalyst-optimized,
        # no Python serialization overhead.
        .withColumn(
            "price_tier",
            when(col("price") >= 400, "PREMIUM")
            .when(col("price") >= 200, "MID")
            .otherwise("BUDGET")
        )

        # Drop the ingestion metadata column — Silver consumers don't need it.
        # Keeps the schema clean for downstream JOINs and aggregations.
        .drop("_ingested_at")
    )
Why Liquid Clustering over partitioning: Static partitions (PARTITIONED BY category) work well when cardinality is low and query patterns are fixed. But if query patterns change or cardinality grows, you're rewriting the table. Liquid Clustering lets you change the clustering keys without a full rewrite — idempotent and adaptive. For a learning pipeline with 5 categories and evolving query patterns, Liquid Clustering is the right default.
GOLD LAYER ↓
04 · Gold
Business Aggregates — Materialized View

Goal

Produce a business-ready aggregation — category-level summary with product count, average price, and tier breakdown. Gold tables are what BI tools and dashboards query. They should be fast, pre-aggregated, and never require the end user to write GROUP BY queries.

Why Materialized View, Not @dp.table

  • @dp.table = streaming table. Requires a streaming source. Aggregations on streams require stateful operations (window functions, watermarks) — complex and resource-intensive
  • @dp.materialized_view = batch. Full refresh on each pipeline run. For a category summary that needs to reflect the latest complete Silver state, a full refresh is correct — a streaming aggregate would produce partial results mid-run
  • Gold aggregations are by nature not incremental — you always want the final number, not a running total
  • Swapping @dp.materialized_view with @dp.table here is the #1 exam trap for this section
pipeline_products.py · Gold Layer
# ── GOLD LAYER ────────────────────────────────────────────────────────
# Business-ready aggregation. Full refresh on every pipeline run.
# BI dashboards and SQL Warehouses query this table — it must be fast.
#
# IMPORTANT: @dp.materialized_view — NOT @dp.table.
# Gold aggregations need the full Silver state, not a streaming partial.
# @dp.table requires a streaming source; GROUP BY on a stream needs
# stateful windowing which is complex and expensive for this use case.

@dp.materialized_view(
    name="gold_category_summary",
    comment="Category-level product summary — primary Gold table for dashboards"
)
def gold_category_summary():
    return spark.sql("""
        SELECT
            category,

            -- Core business metrics
            COUNT(*)       AS product_count,
            ROUND(AVG(price), 2) AS avg_price,
            MIN(price)     AS min_price,
            MAX(price)     AS max_price,

            -- Tier distribution — how many products in each price band?
            -- Useful for category mix analysis without joining back to Silver
            COUNT(CASE WHEN price_tier = 'PREMIUM' THEN 1 END) AS premium_count,
            COUNT(CASE WHEN price_tier = 'MID'     THEN 1 END) AS mid_count,
            COUNT(CASE WHEN price_tier = 'BUDGET'  THEN 1 END) AS budget_count,

            -- Pipeline metadata — when did this Gold row last refresh?
            current_timestamp() AS last_refreshed_at

        FROM databricks_clar.silver.products_enriched
        GROUP BY category
        ORDER BY avg_price DESC
    """)
Reading from the UC table path, not pipeline alias: The Gold materialized view reads from databricks_clar.silver.products_enriched using the full three-part Unity Catalog name rather than the in-pipeline alias "silver_products_enriched". Either works inside a pipeline, but the full path is more explicit and works correctly if the pipeline is split across files later.
AUTO CDC LAYER ↓
05 · Auto CDC
Change Data Capture — SCD Type 1

Goal

Handle product updates (price changes, category corrections) from a CDC-enabled source. Auto CDC processes INSERT, UPDATE, and DELETE operations from the source and applies them as upserts to the Silver target. The result is always the current state of the product — no duplicates.

SCD1 vs SCD2 Decision

  • SCD Type 1 (chosen): overwrite. Product ID 5 gets a new price → old price is gone. Use when you need current state only — dashboards, operational reporting
  • SCD Type 2: keep history. Product ID 5 price change → new row added with effective dates. Use when you need "what was the price last Tuesday?" — compliance, audit, ML feature engineering
  • For this pipeline, we chose SCD1 — Gold already aggregates current prices. Historical pricing would require SCD2 on a separate table
pipeline_products.py · Auto CDC
# ── AUTO CDC — SCD TYPE 1 ─────────────────────────────────────────────
# Handles: INSERT, UPDATE, DELETE from a CDC-enabled source.
# SCD1 = always current state. Old values are overwritten, not kept.
# Use SCD2 (stored_as_scd_type=2) when you need historical record.
#
# create_auto_cdc_flow() replaces the old apply_changes() in legacy DLT.
# The function name changed with the Lakeflow rebrand.

# First: declare the target table (empty shell — CDC will populate it)
@dp.table(name="silver_products_current",
          comment="Current state of all products — maintained by Auto CDC")
def silver_products_current(): pass

# Then: wire up the CDC flow that feeds the target table above
dp.create_auto_cdc_flow(

    # source: the CDC-enabled Bronze table (with _change_type column)
    # This is a STRING — not a function call, not col().
    # Common mistake: wrapping in col() like sequence_by below.
    source="bronze_products_cdc",

    # target: the table declared above — also a string
    target="silver_products_current",

    # keys: the column(s) that uniquely identify a record.
    # Matches + same key → UPDATE. New key → INSERT. Deleted key → DELETE.
    keys=["product_id"],

    # sequence_by: tells CDC which record is "newer" when conflicts arise.
    # MUST be col() — not a string. "processDate" as a string = runtime error.
    # The column must be monotonically increasing (timestamp, version number).
    sequence_by=col("processDate"),

    # No stored_as_scd_type = SCD1 (default).
    # Add stored_as_scd_type=2 for full history with __START_AT/__END_AT cols.

    # except_column_list: exclude CDC metadata columns from the target table.
    # _change_type is CDC plumbing — Silver consumers shouldn't see it.
    # Note: it's except_column_list, NOT except_columns (common exam typo).
    except_column_list=["_change_type", "_commit_timestamp"]
)
sequence_by trap: This is the most commonly missed syntax detail. sequence_by=col("processDate") is correct. sequence_by="processDate" is a runtime error. The col() wrapper is required — it creates a Column expression, not just a string name.
MERGE INTO (PySpark alternative) ↓
06 · MERGE INTO
Manual Upsert — PySpark MERGE (Non-Pipeline Context)

Goal

Demonstrate the PySpark MERGE INTO pattern — the manual alternative to Auto CDC when you're working outside a Lakeflow pipeline (standalone notebook, scheduled job). MERGE gives you full control over the upsert logic at the cost of more code to maintain.

Auto CDC vs Manual MERGE

AspectAuto CDCMERGE INTO
Context✓ Inside pipelineNotebook / Job
Code volume✓ ~10 lines~25 lines
SCD2 support✓ Built-inManual, complex
FlexibilityPipeline-bound✓ Runs anywhere

notebook_merge_products.py · Standalone MERGE
# ── MERGE INTO (PySpark) ──────────────────────────────────────────────
# Use this pattern in: standalone notebooks, LakeFlow Job tasks,
# or any context OUTSIDE a Lakeflow Declarative Pipeline.
# Inside a pipeline: use create_auto_cdc_flow() instead — less code, more features.
#
# Pattern: check if Silver table exists first.
# First run: write (table doesn't exist yet).
# Subsequent runs: MERGE (upsert only changed rows).

from delta.tables import DeltaTable

# Load the incoming batch of changes
df_updates = (
    spark.read.format("csv")
    .option("header", "true")
    .option("inferSchema", "true")
    .load("/Volumes/databricks_clar/bronze/bronze_volume/products/")
    .withColumn("category", upper(col("category")))
    .withColumn("price", col("price").cast(DoubleType()))
)

# TARGET TABLE NAME — using three-part Unity Catalog path
target_table = "databricks_clar.silver.products_enriched"

if spark.catalog.tableExists(target_table):

    # Table exists → MERGE (upsert pattern)
    # DeltaTable.forName() loads the existing Delta table as a DeltaTable object.
    # This is required for the .merge() API — spark.table() returns a DataFrame,
    # which doesn't have the .merge() method.
    target = DeltaTable.forName(spark, target_table)

    (
        target.alias("trg")
        .merge(

            # Source DataFrame aliased as "src"
            df_updates.alias("src"),

            # Match condition — how do we identify the same record?
            # product_id is our business key — must be unique in Silver.
            "trg.product_id = src.product_id"
        )

        # Matched row (product exists) → update ALL columns from source.
        # updateAll() is shorthand for mapping every column by name.
        # Use updateExpr({"price": "src.price"}) for selective column updates.
        .whenMatchedUpdateAll()

        # No match (new product) → insert the row into Silver.
        # insertAll() maps all source columns to target columns by name.
        .whenNotMatchedInsertAll()

        # Execute the MERGE — this is when the actual write happens.
        .execute()
    )

else:
    # First run — table doesn't exist yet.
    # Write the initial dataset as a new Delta table.
    # Mode "append" on a fresh table = same as overwrite for the first run,
    # but "append" is safer if the table creation races with another writer.
    (
        df_updates.write
        .format("delta")
        .mode("append")
        .saveAsTable(target_table)
    )
DeltaTable.forName() vs spark.table(): This is a common point of confusion. spark.table("table_name") returns a DataFrame — read-only. DeltaTable.forName(spark, "table_name") returns a DeltaTable object that exposes the .merge(), .delete(), and .update() APIs. You need the DeltaTable object for any mutating operation.
07 · Summary
Design Decisions at a Glance
Pipeline Design Decisions
Layer / Pattern Chosen Approach Rejected Alternative Reason
Bronze ingestion Auto Loader (cloudFiles) COPY INTO Streaming, incremental, exactly-once. COPY INTO is batch/manual
Bronze trigger availableNow=True Continuous stream Processes queued files then stops — saves compute vs 24/7 stream
Bronze quality @dp.expect (WARN) @dp.expect_or_fail Bronze captures everything. Violations are flagged, not discarded
Intermediate transform Staging view (no storage) Additional Bronze table Zero storage cost. Transforms too simple to justify materialization
Silver quality @dp.expect_or_drop @dp.expect (WARN) Silver is the quality contract. Bad rows dropped, pipeline continues
Silver clustering Liquid Clustering (category, price) PARTITION BY category Idempotent, flexible key changes. Static partitions require full rewrite to change
Gold table type @dp.materialized_view @dp.table (streaming) Aggregations need full Silver state. Streaming aggregates produce partial results
CDC pattern create_auto_cdc_flow() Manual MERGE INTO Inside pipeline: Auto CDC is ~10 lines vs ~25 for MERGE. SCD2 built-in
CDC type SCD Type 1 (current state) SCD Type 2 (history) Dashboards need current prices. Historical pricing = separate SCD2 table
Gold column calcs when/otherwise (SQL-native) Python UDF SQL-native runs in Catalyst optimizer. UDFs serialize to Python = slower
