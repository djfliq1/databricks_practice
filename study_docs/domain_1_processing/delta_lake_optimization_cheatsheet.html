Core Commands
OPTIMIZE
-- Compact small files
OPTIMIZE databricks_clar.silver.products_enriched;

-- With Z-ORDER on query columns
OPTIMIZE databricks_clar.silver.products_enriched
ZORDER BY (category, price);
  • Bin-packing = default, merges small files into ~1GB targets
  • Idempotent — safe to run repeatedly
  • Does NOT delete old files — need VACUUM for that
VACUUM
-- Remove files older than retention
VACUUM databricks_clar.silver.products_enriched;

-- Custom retention (default = 7 days)
VACUUM databricks_clar.silver.products_enriched
RETAIN 168 HOURS;

-- DRY RUN first!
VACUUM databricks_clar.silver.products_enriched DRY RUN;
  • Deletes files not referenced by current/recent versions
  • 7-day minimum to preserve time travel
  • Going below 7 days breaks streaming readers
ANALYZE TABLE
-- Collect stats for query optimizer
ANALYZE TABLE
  databricks_clar.silver.products_enriched
COMPUTE STATISTICS;

-- Column-level stats
ANALYZE TABLE
  databricks_clar.silver.products_enriched
COMPUTE STATISTICS FOR COLUMNS
  product_id, category, price;
  • Feeds Catalyst optimizer with data distribution info
  • Enables better join strategies (broadcast vs shuffle)
Liquid Clustering vs Z-ORDER
Comparison
AspectZ-ORDERLiquid Clustering
SyntaxOPTIMIZE ... ZORDER BY (col)CLUSTER BY (col) at table creation
Idempotent✗ NOT idempotent✓ Idempotent
FlexibilityRewrite entire table to changeChange clustering keys without rewrite
PartitioningWorks with partitioned tablesReplaces partitioning — don't combine
When to useLegacy tables, existing partitioned tablesNew tables — preferred 2026 pattern
Exam signal"which is NOT idempotent""modern replacement for partitioning"
-- Liquid Clustering — define at creation
CREATE TABLE databricks_clar.silver.products_enriched (
  product_id INT, product_name STRING,
  category STRING, price DOUBLE
) CLUSTER BY (category, price);

-- Then just run OPTIMIZE — no ZORDER needed
OPTIMIZE databricks_clar.silver.products_enriched;

-- Query by version number
SELECT * FROM databricks_clar.silver.products_enriched
VERSION AS OF 3;

-- Query by timestamp
SELECT * FROM databricks_clar.silver.products_enriched
TIMESTAMP AS OF '2026-04-01 00:00:00';

df = spark.read.format("delta") \
  .option("versionAsOf", 3) \

Clone Operations
DEEP CLONE — Full Copy
-- Deep clone = full data copy
CREATE TABLE databricks_clar.silver.products_backup
DEEP CLONE databricks_clar.silver.products_enriched;

-- Re-run = incremental (only new changes)
CREATE OR REPLACE TABLE databricks_clar.silver.products_backup
DEEP CLONE databricks_clar.silver.products_enriched;
  • Fully independent — source VACUUM safe
  • Re-running is incremental (copies delta only)
  • Use for: backups, DR, production snapshots
SHALLOW CLONE — Pointer Only
-- Shallow clone = metadata + log only
CREATE TABLE databricks_clar.silver.products_test
SHALLOW CLONE databricks_clar.silver.products_enriched;

-- ⚠ DANGER: if source is VACUUMed,
-- shallow clone files disappear
  • Zero data copy — just pointers to source files
  • Source VACUUM = shallow clone breaks
  • Use for: dev/test only, short-lived experiments
Predictive Optimization
Auto OPTIMIZE + VACUUM — Managed Tables Only
-- Enable on Unity Catalog managed table
ALTER TABLE databricks_clar.silver.products_enriched
SET TBLPROPERTIES ('delta.enablePredictiveOptimization' = 'auto');

-- Or enable at catalog/schema level
ALTER CATALOG databricks_clar
SET TBLPROPERTIES ('delta.enablePredictiveOptimization' = 'auto');
  • Managed Unity Catalog tables ONLY — external tables do not qualify
  • Databricks automatically runs OPTIMIZE and VACUUM based on usage patterns
  • No manual scheduling needed — hands-off maintenance
Exam Traps
⚠ Trap 01
OPTIMIZE ≠ delete old files. OPTIMIZE compacts files but does NOT remove the old ones. You still need VACUUM to reclaim storage.
⚠ Trap 02
Z-ORDER is NOT idempotent. Running ZORDER BY multiple times rewrites data each time with no additional benefit. Liquid Clustering IS idempotent.
⚠ Trap 03
VACUUM below 7 days breaks streaming. The 7-day default isn't arbitrary — going below it means active stream checkpoints may reference deleted files.
⚠ Trap 04
Shallow clone + VACUUM = disaster. If the source table is VACUUMed, shallow clone data files disappear. Deep clone is the safe backup pattern.
⚠ Trap 05
VERSION AS OF ≠ RESTORE. VERSION AS OF is a read-only time travel query. RESTORE TABLE actually rolls the table back. Know the difference.
✓ Key Point
Predictive Optimization = Unity Catalog managed tables only. External tables, even in UC, do not get auto OPTIMIZE/VACUUM.
✓ Key Point
Liquid Clustering replaces partitioning. Do NOT use PARTITION BY and CLUSTER BY on the same table — they are mutually exclusive patterns.
✓ Key Point
OPTIMIZE on a streaming source table is safe. It does not interrupt active streams reading from that table.
