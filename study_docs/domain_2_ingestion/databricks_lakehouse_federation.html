Databricks Lakehouse Federation

Data Engineer Associate Certification Guide

What is Lakehouse Federation?

Query external databases (MySQL, Postgres, Snowflake, Redshift, BigQuery, SQL Server, etc.) from Databricks without ingesting the data. Unity Catalog treats the foreign source as if it were native tables.

Why it matters (exam angle): It's the "no-ETL" pattern — governance, lineage, and access control via Unity Catalog across sources you don't own.

Architecture Overview

The Three Core Objects

1. Connection

What: Stores credentials and connection details

Purpose: Secure authentication path to external database

Governed: Access controlled through Unity Catalog permissions

2. Foreign Catalog

What: Mirrors the remote database schema in Unity Catalog

Purpose: Makes external tables appear as native Databricks tables

Access: Read-only for most operations

3. Query Engine

What: Handles query execution and optimization

Features: Query pushdown, federation, no data movement

Benefits: Maintains lineage and applies ACLs

Setup Workflow (Cert-Relevant SQL)

1

Create the Connection

Stores credentials, governed by Unity Catalog

SQL - Create Connection
CREATE CONNECTION mysql_prod TYPE mysql OPTIONS ( host 'mysql.prod.internal', port '3306', user secret('db_scope','mysql_user'), password secret('db_scope','mysql_pw') );
2

Create the Foreign Catalog

Mirrors the remote DB into Unity Catalog

SQL - Create Foreign Catalog
CREATE FOREIGN CATALOG mysql_sales USING CONNECTION mysql_prod OPTIONS (database 'sales_db');
3

Query Like Native Tables

Use standard SQL syntax, no special federation commands needed

SQL - Federated Query
-- Straight federated query, no ingestion SELECT customer_id, SUM(amount) AS total_spend FROM mysql_sales.public.orders WHERE order_date >= '2026-01-01' GROUP BY customer_id;

Integration with Your Bronze Layer

Your project has a bronze layer CSV with products data. Here's how Federation would let you join it with external inventory data without moving anything:

PySpark - Joining Bronze Layer with Federated Data
# Your bronze layer (from project CSV) bronze_df = spark.read.option("header", "true").csv("/path/to/2_bronze_PySpark.csv") # Join with federated MySQL inventory (no data movement!) federated_query = """ SELECT b.product_id, b.product_name, b.category, b.price, m.stock_count, m.warehouse_location FROM bronze.products_csv b JOIN mysql_sales.public.inventory m ON b.product_id = m.product_id WHERE b.category IN ('Electronics', 'Toys') """ result_df = spark.sql(federated_query) result_df.display()
SQL - Grant Access Through Unity Catalog
GRANT USE CATALOG ON CATALOG mysql_sales TO `data_analysts`; GRANT SELECT ON mysql_sales.public.orders TO `data_analysts`; GRANT SELECT ON mysql_sales.public.inventory TO `data_analysts`;

Certification Exam Cheat Sheet

Concept Remember This
Purpose Query external data without ETL
Governance Unity Catalog (access control, lineage, audit)
Key objects Connection → Foreign Catalog → Tables
Supported sources MySQL, Postgres, Snowflake, Redshift, BigQuery, SQL Server, Databricks-to-Databricks
Performance Uses query pushdown — filters/aggregates execute on source when possible
Writes Mostly read-only (some sources support limited DML)
vs Ingestion Federation = zero-copy; Ingestion = data lands in Delta
vs Delta Sharing Federation = pull from other systems; Delta Sharing = share Delta tables out

🪟 The Mental Model

Federation is a window into another database. Ingestion is a copy into yours.

The exam loves to test this distinction. Remember: Federation maintains data in its original location while providing unified access through Unity Catalog.

Advanced Example: Complex Federated Query

SQL - Multi-Source Federation with Aggregation
-- Complex query joining bronze layer, federated MySQL, and Snowflake WITH product_metrics AS ( SELECT b.product_id, b.product_name, b.category, m.stock_count, s.sales_last_30d, CASE WHEN m.stock_count < 10 AND s.sales_last_30d > 100 THEN 'HIGH_PRIORITY_RESTOCK' WHEN m.stock_count < 50 THEN 'LOW_STOCK' ELSE 'NORMAL' END as inventory_status FROM bronze.products_csv b JOIN mysql_prod.inventory.current_stock m ON b.product_id = m.product_id JOIN snowflake_analytics.sales.product_summary s ON b.product_id = s.product_id WHERE b.processDate >= current_date() - INTERVAL 1 DAY ) SELECT category, inventory_status, COUNT(*) as product_count, SUM(sales_last_30d) as total_sales, AVG(stock_count) as avg_stock FROM product_metrics GROUP BY category, inventory_status ORDER BY total_sales DESC;

Key Performance Considerations

Query Pushdown

Filters, aggregations, and joins are pushed to the source database when possible

Benefit: Reduces network traffic and improves performance

Network Latency

Queries depend on network connection to external sources

Best Practice: Use for analytical queries, not high-frequency transactional workloads

Source Performance

Federation queries impact the source database performance

Consideration: Coordinate with source system owners on query patterns

💡 Exam Tip

Remember the difference between Federation (query external data in place) and Delta Sharing (share your Delta tables with external parties). Both are Unity Catalog features but solve opposite data sharing scenarios!
