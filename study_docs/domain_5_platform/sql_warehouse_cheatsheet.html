Warehouse Types
Serverless
  • Instant start — no cluster provisioning
  • Remote Result Cache — survives restarts
  • No compute config needed
  • Best for: ad-hoc, dashboards, BI
  • Higher per-DBU cost, lower ops overhead
Pro
  • Optimized for BI and dashboards
  • Photon engine enabled
  • Query result caching
  • Supports alerts + scheduled queries
  • Best for: production BI workloads
Classic
  • Legacy type — being phased out
  • No Photon acceleration
  • No Remote Result Cache
  • Slower startup than Pro/Serverless
  • Avoid for new workloads

Query Scheduling & Alerts
Query Scheduling
  • Set directly in SQL Editor per query
  • Runs on a schedule — cron-style or interval
  • Independent from alert schedules
  • Requires the query to have a SQL Warehouse assigned
  • Results refresh the result cache automatically
-- Schedule via UI or API
-- Schedule != alert — they are separate systems
SQL Alerts
  • Alert = Query + Condition + Schedule + Destination
  • Alert schedule and query schedule are independent
  • Trigger types: value, threshold, comparison
  • Destinations: email, Slack, PagerDuty, webhook
  • Requires CAN MANAGE on the warehouse to edit
-- Alert example: notify when avg price drops below 100
SELECT AVG(price) as avg_price
FROM databricks_clar.silver.products_enriched;
-- Then set condition: avg_price < 100 → trigger alert

Exam Traps
⚠ Trap 01
Alert schedule ≠ query schedule. They are completely independent systems. An alert can fire on its own schedule regardless of when the underlying query last ran.
⚠ Trap 02
Dynamic functions break caching. CURRENT_TIMESTAMP(), RAND(), UUID() in a query prevent result caching because the result changes every run.
⚠ Trap 03
Remote Result Cache = Serverless only. This is the primary exam differentiator for Serverless warehouses. Pro and Classic do NOT have it.
⚠ Trap 04
Alerts require CAN MANAGE, not just CAN USE. To create or edit alerts, you need CAN MANAGE on the warehouse, not just query execution permissions.
✓ Key Point
Serverless = fastest startup + Remote Result Cache. In any exam scenario asking "which warehouse type should we use for low-latency ad-hoc queries" — Serverless is the answer.
✓ Key Point
Query Profile is for individual query tuning. Look for: Scan operations (reading too much data?), Sort/Shuffle (expensive operations?), Join type (broadcast vs shuffle?). These are the three profiling operations the exam tests.
