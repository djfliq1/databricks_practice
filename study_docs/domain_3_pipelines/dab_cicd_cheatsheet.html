Databricks Asset Bundles CI/CD

databricks_clar · catalog · bronze → silver → gold
EXAM DOMAIN: Deployment & CI/CD
Cortana TL;DR
  • DAB = Infrastructure as Code for Databricks. One YAML bundle defines jobs, pipelines, clusters, permissions — version-controlled and deployed like software.
  • CI/CD = the automation wrapper. Git push → GitHub Actions → databricks bundle deploy runs automatically per environment target.
  • Environments are targets in the YAML. dev, staging, prod — each gets its own catalog, schema, cluster, or workspace config via variable substitution.
  • databricks.yml is the root. Everything — jobs, DLT pipelines, notebooks, permissions — hangs off this one file. Sub-configs can be split into includes.
  • Key CLI commands: init → validate → deploy → run → destroy. That's the full lifecycle.
CI/CD Loop
Full cycle — code change → feedback → fix → repeat
YAML Structure
databricks.yml — Root Structure
bundle:
  name: clar_products_pipeline

include:
  - resources/*.yml

variables:
  catalog_name:
    default: databricks_clar

targets:
  dev:
    mode: development
    default: true
    variables:
      catalog_name: dev_clar
  prod:
    mode: production
    variables:
      catalog_name: databricks_clar
resources/jobs.yml — Job Definition
resources:
  jobs:
    bronze_ingest_job:
      name: Bronze Ingest — Products
      tasks:
        - task_key: autoloader_task
          notebook_task:
            notebook_path: ./notebooks/bronze
      schedule:
        quartz_cron_expression: "0 0 * * * ?"
        timezone_id: America/New_York
      job_clusters:
        - job_cluster_key: main
          new_cluster:
            spark_version: "15.4.x-scala2.12"
            num_workers: 2

Environments & Targets
dev target
  • mode: development — auto-prefixes resource names with username
  • Avoids collision with prod resources
  • default: true → runs without -t flag
  • Usually maps to dev catalog / dev schema
  • Clusters: smaller / cheaper configs ok
staging target
  • mode: production but pointing to staging workspace
  • Used for integration tests pre-prod
  • Mirrors prod config as closely as possible
  • Triggered by merge to main branch
  • Shared environment — PR gating needed
prod target
  • mode: production — no auto-prefixing
  • Requires manual approval or protected branch
  • Service principal auth (not personal token!)
  • Maps to databricks_clar catalog
  • Destroy commands blocked / gated
Git & Databricks Repos
Databricks Repos (Git Folders)
  • Repos = Git integration natively inside Databricks workspace
  • Sync notebooks directly from GitHub/GitLab/ADO/Bitbucket
  • Branch, commit, PR from within workspace UI
  • DAB references notebook paths relative to repo root
  • Each dev can work on their own branch simultaneously
  • Not required for DAB — but complements it for notebook-heavy workflows
GitHub Actions — CI Pipeline
# .github/workflows/deploy.yml
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: databricks/setup-cli@main
      - run: databricks bundle validate
      - run: databricks bundle deploy -t prod
        env:
          DATABRICKS_HOST: ${{ secrets.DB_HOST }}
          DATABRICKS_TOKEN: ${{ secrets.DB_TOKEN }}
DLT Pipelines Inside a Bundle
resources/pipelines.yml — DLT via DAB
resources:
  pipelines:
    silver_products_dlt:
      name: Silver Products Pipeline
      target: ${var.catalog_name}.silver
      catalog: ${var.catalog_name}
      development: false
      libraries:
        - notebook:
            path: ./notebooks/silver_dlt
      configuration:
        pipelines.enableTrackHistory: "true"
  • libraries — point to notebooks or Python files containing @dlt.table decorators
  • target — schema where DLT writes tables in Unity Catalog
  • catalog — Unity Catalog catalog name (uses variable)
  • development: false — set true in dev target to use DLT dev mode (no data published)
  • configuration — pipeline-level Spark/DLT configs injected here
  • DLT pipelines can be triggered by a Job task defined in jobs.yml
How DAB Tracks Changes
State File — The Source of Truth
  • DAB stores a state file in the workspace (DBFS or workspace FS)
  • Records all deployed resource IDs per target
  • On next deploy → compares current YAML vs state
  • Adds new resources, updates changed ones, removes deleted ones
  • State file = why destroy knows what to tear down
  • Each target has its own independent state
dev mode Auto-Prefix Behavior
  • In mode: development, Databricks prepends [dev username] to resource names
  • bronze_ingest_job → [dev clarence] bronze_ingest_job
  • Prevents your dev job colliding with coworker's or prod
  • Also prefixes DLT pipeline names
  • Does NOT affect catalog/schema — you control those with variables
  • Prod mode = exact names, no prefix
Exam Traps
⚠ Trap 01 — destroy ≠ data delete
bundle destroy removes Databricks workspace objects (jobs, pipelines, clusters) — it does NOT drop Delta tables or delete files in Unity Catalog. Your data survives.
⚠ Trap 02 — validate does not deploy
bundle validate only checks YAML syntax and config references. No resources are created or modified. Safe to run without workspace write permissions.
⚠ Trap 03 — dev mode prefix is automatic, not configurable
You cannot change the prefix format in dev mode. The [dev username] prefix is injected by Databricks automatically. If you set mode: development, it happens — full stop.
⚠ Trap 04 — personal token vs service principal
In prod CI/CD (GitHub Actions, ADO), you must use a service principal token, not a personal access token. Personal tokens are tied to individual users and break when that person leaves.
⚠ Trap 05 — include paths are relative to databricks.yml
If your databricks.yml is in the root and you write include: resources/*.yml, that path must resolve from the root directory. Wrong relative path = silent resource exclusion.
✓ High-Value Point — Variable substitution syntax
DAB variables use ${var.variable_name} syntax — NOT Jinja, NOT double curly. If you see {{ catalog }} that's wrong for DAB. Correct: ${var.catalog_name}.
✓ High-Value Point — bundle run requires prior deploy
You can't bundle run a resource that hasn't been deployed first. Run operates on already-deployed objects in the workspace. The sequence is always: deploy → run.
