LakeFlow Jobs

formerly Databricks Workflows · databricks_clar
EXAM DOMAIN: Productionizing 18%
Cortana TL;DR
  • Job = DAG of tasks — tasks run in dependency order, each task is a unit of work
  • Task types: Notebook, Python Script, SQL, DLT Pipeline, For Each, Spark Submit
  • Task values = pass data between tasks via dbutils.jobs.taskValues
  • Compute options: Job cluster (recommended), All-purpose cluster, Serverless
  • Triggers: Scheduled (cron), File arrival, Continuous, Manual
Task Types
Notebook Task
# Pass parameters into notebook
dbutils.widgets.get("category")

# Set task value output
dbutils.jobs.taskValues.set(
  key="record_count",
  value=df.count()
)
  • Most flexible — full PySpark/SQL support
  • Can read/write task values
SQL Task
-- Run SQL against a warehouse
SELECT
  category,
  COUNT(*) as cnt,
  AVG(price) as avg_price
FROM databricks_clar.silver.products_enriched
GROUP BY category
  • Runs on SQL Warehouse (not cluster)
  • Output: first row as SQL task output
For Each Task
# For Each iterates over a list
# Input: JSON array from prior task
# Iterates: runs child task per item

# In child notebook:
item = dbutils.widgets.get("item")
# item = each element of the array
  • Parallel execution of child task
  • Great for per-category processing
Task Values — Pass Data Between Tasks
dbutils.jobs.taskValues
# Task A: Set a value
result = databricks_clar.silver.products_enriched
record_count = spark.sql("SELECT COUNT(*) FROM databricks_clar.silver.products_enriched").collect()[0][0]
dbutils.jobs.taskValues.set(key="record_count", value=record_count)

# Task B: Read the value from Task A
count = dbutils.jobs.taskValues.get(
  taskKey="task_a",
  key="record_count",
  default=0,
  debugValue=100
)

# Reference in YAML / UI with dynamic values syntax
# {{tasks.task_a.values.record_count}}
  • debugValue — returned when running notebook interactively (not in job context)
  • default — returned if the key doesn't exist
  • Values must be JSON-serializable (string, number, list, dict)
  • Dynamic values syntax in YAML: {"{{tasks.task_name.values.key}}"}
Triggers & Schedules
Four Trigger Types
TriggerWhen It FiresUse Case
Scheduled (Cron)Time-based — every N minutes, daily, weeklyNightly ETL, hourly refresh, weekly reports
File ArrivalNew file lands in a Volume or cloud storage pathEvent-driven ingestion — process when data arrives
ContinuousRuns immediately after prior run completesNear-real-time processing with minimal gap
ManualUser-triggered via UI, API, or CLIOne-off runs, testing, on-demand backfill
# Quartz cron in YAML
schedule:
  quartz_cron_expression: "0 0 6 * * ?"  # 6am daily
  timezone_id: "America/New_York"
Compute Selection
Job Cluster (Recommended)
  • Created at job start, terminated at end
  • Pay only while job runs
  • Config locked per job — consistent runs
  • Exam default answer for scheduled jobs
All-Purpose Cluster
  • Always-on, shared across users
  • More expensive — running 24/7
  • OK for interactive dev, not prod jobs
  • Risk: cluster config may change unexpectedly
Serverless
  • Instant start — no provisioning wait
  • Managed by Databricks
  • Best for: short jobs, fast iteration
  • Higher per-DBU cost, zero ops overhead
Exam Traps
⚠ Trap 01
All-purpose clusters are expensive for scheduled jobs. The exam-correct answer for nightly ETL compute is always Job Cluster — created on trigger, terminated on completion.
⚠ Trap 02
Task values ≠ notebook widgets. dbutils.jobs.taskValues passes data between tasks. dbutils.widgets passes parameters into a notebook. Different systems, different purposes.
⚠ Trap 03
For Each requires a JSON array input. The input to For Each must be a JSON array (from a prior SQL Rows task or task value). It can't iterate over arbitrary data types.
✓ Key Point
Dynamic values syntax: {"{{tasks.task_name.values.key_name}}"} — this is how you reference task outputs in YAML or other task configs. The double-curly syntax is specific to LakeFlow Jobs dynamic values.
✓ Key Point
File arrival trigger = event-driven pipeline. Set a path on a Unity Catalog Volume and the job fires whenever a new file lands there. No polling, no cron — pure event-driven.
