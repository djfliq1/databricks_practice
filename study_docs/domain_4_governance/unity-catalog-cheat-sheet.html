🎯 Core Concepts

What is Unity Catalog?

Unity Catalog is a centralized data governance solution that provides unified metadata, security, and discovery across all Databricks workspaces in a region.

Key Benefits:
  • Single metastore per region serves multiple workspaces
  • Three-level namespace: catalog.schema.table
  • ANSI SQL GRANT/REVOKE permissions
  • Automatic lineage tracking
  • Cross-workspace data access without duplication

🏗️ Architecture Overview

Unity Catalog Architecture

The metastore is the centralized metadata repository that holds all information about your data assets. One metastore per region serves multiple workspaces, creating unified governance across your entire Databricks environment.

🗂️ Namespace Structure

Three-Level Hierarchy

SQL
-- Full qualified name
SELECT * FROM catalog_name.schema_name.table_name;

-- Examples
SELECT * FROM sales.transactions.orders;
SELECT * FROM ml.experiments.feature_table;
SELECT * FROM external.legacy.customer_archive;

Catalog → Schema → Objects

  • Catalog: Top-level container (department/domain)
  • Schema: Logical grouping within catalog (functional area)
  • Objects: Tables, views, functions, volumes, ML models

🌍 Environment Patterns

Catalog-per-Environment

SQL
-- Development
CREATE CATALOG dev_sales;
CREATE SCHEMA dev_sales.transactions;
CREATE TABLE dev_sales.transactions.orders (...);

-- Staging
CREATE CATALOG staging_sales;
CREATE SCHEMA staging_sales.transactions;
CREATE TABLE staging_sales.transactions.orders LIKE dev_sales.transactions.orders;

-- Production
CREATE CATALOG prod_sales;
CREATE SCHEMA prod_sales.transactions;
CREATE TABLE prod_sales.transactions.orders LIKE staging_sales.transactions.orders;

Workspace-per-Environment

  • dev-workspace → Access all three catalogs
  • staging-workspace → Access staging + prod (read-only)
  • prod-workspace → Access prod only

📊 Managed vs External Tables

Aspect Managed Tables External Tables
Data Ownership Unity Catalog owns metadata + data UC manages metadata, you own data
Storage Location UC-managed location Your specified S3/ADLS path
DROP TABLE Effect Deletes metadata + data files Only removes metadata
Recovery 7-day undrop window No undrop available
Optimization Automatic compaction Manual management required

Managed Tables

SQL
-- Create managed table
CREATE TABLE sales.customers.profiles (
    customer_id BIGINT,
    email STRING,
    created_at TIMESTAMP
) -- Data stored in UC-managed location

External Tables

SQL
-- Create external table
CREATE TABLE sales.customers.external_profiles (
    customer_id BIGINT,
    email STRING,
    created_at TIMESTAMP
)
LOCATION 's3://my-bucket/customer-data/'
⚠️ Important: When you DROP TABLE on a managed table, both the metadata AND the actual data files are deleted (with a 7-day undrop window). For external tables, only the metadata is removed.

📁 Volumes (Non-Tabular Data Governance)

Volumes extend Unity Catalog governance to non-tabular data like PDFs, images, ML models, and raw logs.

Managed Volumes

SQL
-- UC creates and manages the storage location
CREATE VOLUME ml.experiments.model_artifacts;

-- Files automatically organized under:
-- /metastore-root/ml/experiments/model_artifacts/

External Volumes

SQL
-- Point to existing storage location
CREATE VOLUME sales.analytics.legacy_reports 
LOCATION 's3://my-data-bucket/reports/quarterly/';

🔐 Permissions and Security

SQL GRANT/REVOKE

SQL
-- Catalog-level permissions
GRANT USE CATALOG sales TO `data-analysts`;
GRANT CREATE SCHEMA ON CATALOG sales TO `data-engineers`;

-- Schema-level permissions  
GRANT USE SCHEMA sales.transactions TO `finance-team`;
GRANT CREATE TABLE ON SCHEMA sales.transactions TO `senior-engineers`;

-- Table-level permissions
GRANT SELECT ON TABLE sales.customers.profiles TO `marketing-team`;
GRANT INSERT ON TABLE sales.transactions.orders TO `order-service`;

-- Volume permissions
GRANT READ FILES ON VOLUME ml.experiments.model_artifacts TO `data-scientists`;
GRANT WRITE FILES ON VOLUME ml.experiments.model_artifacts TO `senior-ml-engineers`;

-- Row-level security
CREATE OR REPLACE FUNCTION sales.customers.row_filter(region STRING)
RETURN IF(IS_ACCOUNT_GROUP_MEMBER('region-managers'), TRUE, region = 'US');

ALTER TABLE sales.customers.profiles 
SET ROW FILTER sales.customers.row_filter(region);

External Location Management

SQL
-- Create storage credential
CREATE STORAGE CREDENTIAL s3_cred 
USING SERVICE PRINCIPAL 
OPTIONS (
  'roleArn' = 'arn:aws:iam::123456789:role/unity-catalog-role'
);

-- Create external location
CREATE EXTERNAL LOCATION s3_data_lake
URL 's3://company-data-lake/raw/'
WITH (STORAGE CREDENTIAL s3_cred);

-- Grant access to external location
GRANT READ FILES ON EXTERNAL LOCATION s3_data_lake TO `data-engineers`;

Tagging

Information Schema Queries

SQL
-- Find tables by name pattern
SELECT * FROM information_schema.tables 
WHERE table_name LIKE '%customer%';

-- Find columns containing PII
SELECT * FROM information_schema.columns 
WHERE column_comment LIKE '%email%' OR column_name LIKE '%email%';

-- View table lineage
SELECT * FROM information_schema.table_lineage 
WHERE target_table_name = 'customer_summary';

💡 Best Practices

Naming Conventions

SQL
-- Catalogs: domain/department based
CREATE CATALOG sales;
CREATE CATALOG marketing; 
CREATE CATALOG engineering;

-- Schemas: functional areas within domain
CREATE SCHEMA sales.transactions;
CREATE SCHEMA sales.customers;
CREATE SCHEMA sales.reporting;

-- Tables: descriptive, avoid generic names
CREATE TABLE sales.transactions.order_line_items;  -- Good
CREATE TABLE sales.transactions.data;              -- Bad

Environment Strategy

  1. Catalog-per-environment for complete isolation
  2. Workspace-per-environment for compute isolation
  3. Use consistent naming: {env}_{domain} (e.g., prod_sales, dev_sales)

Security Strategy

  1. Start with least privilege access
  2. Use groups instead of individual user grants
  3. Implement row-level security for sensitive data
  4. Regular audit of permissions using SHOW GRANTS

Data Organization

  1. Use managed tables for new, Unity Catalog-native workflows
  2. Use external tables for existing data lakes during migration
  3. Leverage volumes for ML artifacts and non-tabular data
  4. Implement consistent tagging strategy early

Performance

  1. Use Delta Lake format for all new tables
  2. Implement proper partitioning strategy
  3. Regular OPTIMIZE and VACUUM operations
  4. Monitor query performance through Unity Catalog lineage

🔧 Common SQL Patterns

Data Discovery

SQL
-- Find all tables in a catalog
SHOW TABLES IN sales;

-- Search for tables by pattern
SHOW TABLES IN sales LIKE '*customer*';

-- View table schema
DESCRIBE EXTENDED sales.customers.profiles;

-- Check table properties and metadata
SHOW TBLPROPERTIES sales.customers.profiles;

Cross-Catalog Queries

SQL
-- Join tables from different catalogs
SELECT 
    p.customer_id,
    p.email,
    o.order_total
FROM prod_sales.customers.profiles p
JOIN dev_sales.transactions.orders o 
    ON p.customer_id = o.customer_id;

Time Travel and Versioning

SQL
-- Query table at specific timestamp
SELECT * FROM sales.customers.profiles 
TIMESTAMP AS OF '2024-01-01 00:00:00';

-- Query table at specific version
SELECT * FROM sales.customers.profiles VERSION AS OF 123;

-- View table history
DESCRIBE HISTORY sales.customers.profiles;
